ROSA on MNIST

Memory, memorization, and the limits of statistical tokens

TL;DR

  • We tried to use ROSA (a pure-statistics suffix-automaton next-token predictor from bcml-labs/rosa-plus) as the memory layer of an MNIST/Fashion-MNIST classifier — neural input + ROSA + neural output.
  • On MNIST, ROSA's contribution is small and noisy: +0.15pp over the no-ROSA baseline.
  • On Fashion-MNIST, ROSA is actively ignored by the output module when its predicted-token-embedding is stop-gradient'd — a lesion test gives +0.01pp. Removing the stop_grad lets ROSA contribute exactly +0.10pp — real, but tiny.
  • The most striking finding is the decoupled experiment: train a VQ-AE on images, build ROSA over the resulting tokens, classify by reading off ROSA's distribution at the <LABEL> position. Result on Fashion-MNIST: train 100% / test 54.47%. ROSA perfectly memorizes training streams but generalizes poorly because reconstruction-optimal tokens aren't class-discriminative enough.
  • The takeaway: ROSA's exact-match memory is real, but on image-classification benchmarks the recent-window of token content already encodes the same information ROSA could provide — and the parts ROSA uniquely knows (training-history correlations) get washed out by codebook collisions.

1 · Background — what is ROSA?

ROSA (introduced by BlinkDL as an extension to RWKV) is a pure-statistics next-token predictor built around a suffix automaton (SAM). After ingesting a token stream, ROSA can be queried with any prefix and returns:

  1. SAM-deterministic: the next token after the rightmost training match of the longest suffix of the prefix that exists in the SAM (a literal "1-NN by token-suffix" rule).
  2. Witten–Bell fallback: when no deterministic answer is found, an interpolated n-gram distribution along the suffix-link chain.

ROSA has zero trainable parameters. It's a frozen data structure that grows monotonically as tokens stream in. The bcml-labs/rosa-plus implementation is character-level for English text; we ported it to integer tokens (rosaplus.py → lib/rosa_core.py).

The question we set out to ask: can ROSA's exact-match memory provide a useful prior for visual classification when slotted into a learned tokenization pipeline?

2 · Architecture

The pipeline (experiments1.py):

image (28×28)                                  label (digit 0..9)
   │                                                  │
   ▼                                                  ▼
recurrent input MLP × K_SAMPLE=16 steps     recurrent input MLP × K_LABEL=4 steps
(RMSNorm + GELU + residual)                 (same shared body, different head)
   │                                                  │
   ▼ z (continuous)                                   ▼ z (continuous)
DiVeQ quantize (V=1024, D=64)               DiVeQ quantize
   │ argmin → integer IDs                             │
   ▼                                                  ▼
[ <SAMPLE>=0,  s_1..s_16,  <LABEL>=1,  l_1..l_4 ]    ← 22-token stream
   │
   ▼
ROSA SAM (continual; grows across all training)
   │ predicted next-token ID at every position
   ▼
codebook[predicted_id]  →  rosa_pred_emb (stop_grad in exp1; no stopgrad in exp6)
                              │
   ┌─ 8-token sliding window ─┘
   │
   ▼
output module: 11-class softmax (10 digits + NO_OUTPUT)
   - fires at every position
   - target: digit at LABEL position (index 17), NO_OUTPUT elsewhere
   - λ_digit_boost = 20 to counter ~21:1 imbalance

Stage-1 pipeline (Phase A, scripts/tmp/decoupled_rosa.py) drops the joint loss and trains the encoder + codebook as a pure VQ-AE on images, then builds ROSA over the resulting frozen tokens, then predicts digits by reading ROSA's distribution at <LABEL> with zero trainable parameters in the output head.

4 · Lesion: is ROSA actually contributing?

For each trained joint model (exp1, exp1n, exp6) we re-evaluated on the test set with three different values of rosa_pred_emb:

  • A · zero: rosa_pred_emb = 0 at every position.
  • B · const: rosa_pred_emb = codebook[0] (a fixed indicator-style vector).
  • C · real: ROSA's actual predicted-token embedding, looked up from the saved 3.3 GB SAM.

C − A measures ROSA's actual contribution. ≈0 means the output module ignores ROSA; positive means ROSA contributes; negative means ROSA hurts.

The pattern matches a failure mode the plan explicitly anticipated:

"The output module may learn to ignore ROSA's feature (it's stop-grad'd anyway) and just classify from the recent-window pool — degenerating to a vanilla VQ-MLP classifier."

That's exactly what happens. The W=8 sliding window of token embeddings already contains everything that determines ROSA's prediction (since ROSA is a deterministic function of recent stream + entire training history). The genuinely new information ROSA provides is the training-history correlations, but with stop_grad the output module has no way to ask the codebook to encode that history usefully — so it learns to zero out the ROSA channel.

5 · Phase A — ROSA as the entire classifier (decoupled)

Architecturally we wanted to test the opposite extreme: force ROSA to be the answer, with as little neural assist as possible. Three stages:

  1. VQ-AE on Fashion-MNIST images only (no labels). Recurrent input MLP + codebook + simple decoder, 50 epochs, reconstruction MSE + commitment loss.
  2. Build ROSA. Frozen encoder. Per-sample stream [<SAMPLE>, s_1..s_16, <LABEL>, y_token] where the digit's token ID is reserved (y + 2 ∈ {2..11}). Feed all 60K streams to the SAM, build the Witten–Bell LM.
  3. Eval (zero parameters). Encode test image → 16 tokens → walk SAM along [<SAMPLE>, s_1..s_16, <LABEL>] → ROSA's distribution at the resulting state → sum probability mass over IDs 2..11 → argmax.

Two prediction modes were tested in Stage 3:

ModeTest accTrain accNotes
v1 — SAM-deterministic shortcut41.63%100.00%1-NN-by-suffix; brittle on novel test images
v2 — Witten–Bell distribution argmax54.47%100.00%aggregates counts up the suffix chain

The 100% train / 54% test split is the most interesting result of the whole project. ROSA is doing exactly what it was designed to do — perfectly remember the training stream — and that's why it can recover every training-set digit by direct lookup. But for a novel test image whose token sequence isn't an exact training match, ROSA's suffix-fallback finds some training stream that ends in this prefix, and that training stream's class is the one ROSA returns. With reconstruction-optimal tokens, similar-looking-but-different-class images often share token prefixes, so the wrong class wins.

6 · Discussion

Why doesn't ROSA help in the joint setting? The output module's recent-window of W=8 tokens already encodes everything that determines ROSA's argmax prediction (since ROSA is a deterministic function of recent stream + entire training history). The truly-novel info ROSA carries is "training-history correlations on this exact suffix" — but:

  • With stop_grad, the output module can't shape what those training-history correlations mean;
  • Without stop_grad, it can — but the gradient through codebook[predicted_id] is biased (changing the codebook would change ROSA's choice, but we don't account for that). The result is a real but tiny improvement (+0.10pp).

Why does decoupled ROSA-as-classifier overfit? Because a SAM is a perfect memory device: it can answer "what came after this exact suffix in training?" with literal recall. Train accuracy is 100% by construction — every training stream's <LABEL>-state has its own correct y_token recorded. Test accuracy is whatever the codebook's class-discriminativity allows: with reconstruction-optimal tokens, similar-looking items in different classes share suffixes, so the SAM hands back the wrong class.

The ceiling for "ROSA-pure" on Fashion-MNIST is ~55%. Real but unimpressive against the 89% dense baseline. The 35 percentage points of headroom would have to come from making the tokens more class-discriminative — but as soon as we do that with supervision (a small classification head in Stage 1, e.g.) we're back to a regular classifier with ROSA bolted on.

The honest interpretation: ROSA's exact-match statistics are a real signal — strong enough to lift random tokens from chance to 41% on Fashion-MNIST and to perfectly recover training data — but on image classification benchmarks the W-token sliding window is already capacity-rich enough that ROSA's contribution is largely redundant. ROSA's natural domain is probably discrete sequences with literal repetition (text, code, log streams) where its database-style memory has more unique value. As BlinkDL noted in the original ROSA-Plus README: ROSA is a "database-like" predictor that excels at surface structure — coherent on syntax, brittle on semantics — and image classification on a small alphabet of clustered codebook tokens turns out to be more "semantics" than "surface".

7 · What's next

  • EGGROLL/ES with ROSA in the loop — black-box-optimize the input MLP + codebook on classification accuracy with ROSA inside the loss. Forces the optimizer to discover tokens that make ROSA's predictions accurate. Slower, but architecturally aligns the optimizer with the structural role of ROSA.
  • Pivot the question — if ROSA's natural strength is exact-match memory, image classification was the wrong testbed. A small one-shot/few-shot benchmark, or a sequential-prediction task (next-pixel, next-character) is closer to ROSA's home turf.
  • ROSA-as-logit-prior — use ROSA's full distribution to initialize the output module's logits, with the neural network producing only a residual. ROSA becomes structurally unavoidable; the decoupled finding suggests this would land somewhere between the joint and decoupled numbers.
